01

Executive Summary

The Problem

Agents manually cross-referenced client profiles against listings using dropdown search and memory, slow, imprecise, and impossible to scale. A client saying "something like the last place but quieter and with a pool under 800k" required mentally joining three disconnected data sources. Non-Spanish-speaking agents faced an additional language barrier on top of that.

The Solution

A bilingual NLP search layer that extracts structured intent from natural language queries, cross-references CRM client profiles at runtime, and returns ranked listings with conversational follow-up support, in the agent's language of choice.

Bottom Line

Time-to-shortlist dropped from 20 minutes to 2. Match quality improved sharply, shortlist acceptance rate up 120% versus the dropdown baseline. 98% of agents switched voluntarily within the first weeks post-launch.

02

The Pain Point

Market / User Reality

Agents operated across three disconnected systems: a listing database, a CRM, and their own memory of previous client interactions. Search was entirely filter-driven, explicit criteria only. The gap between how buyers describe properties ("something like the last place but quieter") and how listings are tagged in a database is where the process collapsed.

Non-Spanish-speaking agents querying in English hit an additional wall: either search in the wrong language, or depend on colleagues for translation before even starting. Agents working with foreign buyers were doing informal translation work before touching search.

Business Impact

At 20 minutes per shortlist, a 10-client week costs over 3 hours in search alone, before qualification or follow-up. Language friction compounded this: mistranslated or oversimplified buyer intent produced shortlists that missed the mark and required rework. In a high-turnover coastal market, slow or imprecise matching directly loses deals.

Why AI, Not a Traditional Heuristic

Dropdown filters handle explicit criteria. They cannot interpret relative preferences ("quieter"), bilingual phrasing, or buyer intent that contradicts their own stated filters. Rule-based systems would require an exhaustive ontology of property descriptors plus language detection logic, fragile, expensive to maintain, and still unable to handle ambiguity. The problem is fundamentally about intent translation, which is where language models operate natively.

Solution & PM Strategy

Product Hypothesis

"If agents can describe a client need in plain language and the system cross-references both listings and CRM profiles automatically, then shortlisting time and match quality will improve, because the bottleneck is intent translation and context retrieval, not listing availability."

The Secret Sauce

Claude performs two distinct tasks in sequence: structured extraction (parsing implicit filters from unstructured queries) and response generation (conversational output with follow-up support). Enforcing output schemas on the extraction step was the key reliability decision, it made downstream ranking deterministic regardless of how loosely the agent phrased the query.

Hallucination risk was mitigated by strict retrieval grounding: the model cannot invent properties. Every result is anchored to a retrieved listing. Follow-up queries ("show me cheaper ones," "bigger garden") are resolved in the same session without requiring the agent to re-enter context.

UX / AI Integration Approach

Copilot pattern, conversational input sits above traditional search, not replacing it. Agents stay in control: AI shortlists, agent validates and sends. This kept adoption friction low and trust high. The system doesn't make decisions; it surfaces ranked options and explains why.

Build vs. Buy

Chose Claude via API over fine-tuning a smaller model. Rationale: at this query volume, inference cost was manageable, and the flexibility to handle diverse bilingual phrasing without training data was worth the per-call cost. Pinecone over self-hosted vector search for the same reason, operational simplicity at this scale trumped marginal cost savings.

Hard-Won Lessons

The Pivot

Early prompt designs attempted full filter extraction in a single pass. On edge cases and ambiguous queries, output was unreliable. Moved to a two-step extract-then-confirm flow: Claude first produces a structured filter object, which is reviewed before retrieval runs. Improved extraction consistency significantly at the cost of marginal latency, the right tradeoff for a tool agents use daily.

Ethics & Governance

No PII stored in vector embeddings. CRM profile data is injected at query time as context, not indexed in Pinecone. This keeps GDPR exposure manageable. One open item: query logs, if persisted, require a defined retention policy. Flagged but not yet enforced, worth closing before scale increases.

What I'd Do Differently

Instrument the metrics from day one. The data to prove this worked was always there, time-to-shortlist, acceptance rate, adoption, but we weren't logging it intentionally at kickoff. Two months in, we were reconstructing baselines instead of comparing against them. Every AI project should start with an eval plan, not end with one.
