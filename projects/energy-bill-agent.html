Executive Summary

The Problem

French households overpay on electricity contracts due to opaque billing and no easy way to match real consumption against available market offers. Existing comparison tools require manual data entry and rely on consumption estimates — users input guesses, get generic outputs they don't trust.

The Solution

A three-agent pipeline that extracts bill data via OCR, fetches verified consumption data directly from Enedis, and generates personalized switching recommendations grounded in the user's actual usage profile.

Bottom Line

28% of users who received a recommendation switched provider, saving an average of €38/month — more than double the market baseline of ~€214/year for a typical household switching from EDF Tarif Bleu.

02

The Pain Point

Market Reality

Existing French comparison platforms (HelloWatt, JeanMarcMorin) require manual data entry and rely on consumption estimates. Users input guesses, receive generic outputs, and don't trust the result enough to act. The information asymmetry between suppliers and households is structural — bills are intentionally complex, and switching friction is high.

Cost of Inaction

Switching from EDF Tarif Bleu to the cheapest available offer saves an average household approximately €214/year on standard consumption (4,770 kWh). Most never switch. The problem is not lack of alternatives — it's lack of a trustworthy, low-effort path to act on them.

Why AI, Not Heuristics

Bill formats vary across providers and change over time. A rules-based extractor breaks on layout updates. Mistral OCR with a structured extraction prompt handles format heterogeneity without per-provider maintenance. The recommendation layer must reason across contract type, consumption profile, and live market offers simultaneously — not reducible to a decision tree.

Solution & PM Strategy

The Three-Agent Pipeline
Agent 1
Bill Extraction
Mistral OCR parses the uploaded PDF. Outputs a strict JSON schema. Fields below confidence threshold trigger a user confirmation step — no silent guessing on financial data.
Agent 2
Consumption Fetch
Enedis API call using the PDL number extracted from the bill. Verified consumption data replaces all estimates. Async with explicit loading state to prevent drop-off.
Agent 3
Recommendation
Claude reasons across contract type, real consumption profile, and live market offers. Prompted to explain reasoning per output line — for hallucination mitigation and user trust.
Product Hypothesis

"If we replace manual data entry with automated bill extraction and real consumption data, then provider switch rate will exceed 20%, because users act on recommendations grounded in their actual usage rather than estimates."

UX Integration Approach

Explicit copilot pattern. User uploads bill, reviews extracted data, receives a structured recommendation card. AI output is visible and auditable at every step. An invisible agentic flow was considered and rejected — financial context requires user trust, not frictionless automation. Showing the work is the product.

Build vs. Buy

Mistral OCR over a custom extractor for format heterogeneity without maintenance overhead. Enedis API over user self-reporting — real data was a non-negotiable given the financial stakes of the recommendation. Claude over a cheaper model for the recommendation step — reasoning quality directly affects switch rate, which is the product's only metric that matters.

Qualitative

Showing the extraction step — rather than hiding it — reduced abandonment and increased confidence in the output.

06

Hard-Won Lessons

The Pivot

Mistral OCR performed well on standard EDF and Engie layouts but failed on regional distributor formats — different table structures, multi-column layouts, low-DPI scans. With no eval set, prompt iterations happened reactively in production, one broken format at a time. Most of month 2 was firefighting. Correction rate eventually dropped from 40% to 6%, but the cost was avoidable.

Ethics & Governance

PDL numbers and consumption data are personal data under GDPR. Explicit consent step before any Enedis API call. No model fine-tuning on user data. Bill PDF retention policy post-extraction should be confirmed and enforced before scaling. confirm retention policy

What I'd Do Differently

Build a bill format eval set of 30 to 50 diverse PDFs before writing a single line of OCR prompt. Format variance was discovered in production, the worst place to find it. A structured extraction benchmark upfront would have halved the iteration cycle and prevented month-2 firefighting entirely. Every extraction project should start with a format audit, not end with one.
